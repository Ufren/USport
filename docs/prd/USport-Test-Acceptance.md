# USport 测试与验收方案

文档版本：V1.0  
更新时间：2026-03-22  
适用范围：MVP 测试、验收与上线前检查

## 1. 测试目标

确保以下内容稳定：

- 登录与鉴权
- 用户资料
- 邀约
- 活动与报名
- 签到与完赛
- 举报与信用
- 会员与支付

## 2. 测试分层

### 2.1 单元测试

重点覆盖：

- 状态流转判断
- 参数校验
- 错误码返回
- 工具函数与共享枚举

### 2.2 接口测试

重点覆盖：

- 鉴权成功和失败
- 幂等场景
- 报名和取消
- 邀约处理
- 举报提交流程

### 2.3 端到端测试

重点覆盖：

- 新用户登录并完善资料
- 创建活动并报名
- 签到并完赛
- 举报并后台处理

### 2.4 回归测试

每次发布前必须执行：

- 登录回归
- 活动主链路回归
- 举报治理回归
- 会员支付回归

## 3. 核心验收场景

### 3.1 登录验收

验收点：

- 微信登录成功后返回 token
- 首次登录会创建用户
- 无效 code 会返回正确错误码

### 3.2 资料验收

验收点：

- 基础资料可编辑
- 运动画像可保存
- 用户详情与当前用户接口字段一致

### 3.3 邀约验收

验收点：

- 可发起邀约
- 对方可接受或拒绝
- 非 `pending` 状态不能重复处理
- 拉黑关系下禁止邀约

### 3.4 活动验收

验收点：

- 活动可创建
- 报名截止后不可报名
- 满员状态正确
- 候补逻辑正确
- 活动取消会同步影响报名状态

### 3.5 履约验收

验收点：

- 到场签到成功
- 未签到用户可被标记为未到场
- 完赛后不允许再次签到

### 3.6 举报验收

验收点：

- 举报可提交
- 证据可保存
- 后台可处理结论
- 举报成立后信用分发生变化

### 3.7 支付验收

验收点：

- 订单可创建
- 支付成功后状态正确
- 退款后权益正确回收或保留

## 4. 异常场景清单

必须覆盖：

- Token 过期
- 用户被封禁后继续操作
- 活动创建后被取消
- 报名时活动已满
- 邀约过期后仍尝试接受
- 举报重复提交
- 订单重复支付回调

## 5. 非功能验收

### 5.1 性能

最低要求：

- 活动列表接口在常规分页下响应稳定
- 登录接口无明显超时
- 高峰期报名接口不允许超卖

### 5.2 安全

最低要求：

- 非法 token 无法访问受保护资源
- 越权访问返回正确错误码
- 敏感字段不在无权限接口返回

### 5.3 可观测性

最低要求：

- 错误请求可在日志中追踪
- 核心业务事件有埋点
- 后台处理记录可审计

## 6. 发布前检查清单

### 6.1 研发检查

- 接口文档已更新
- 数据库迁移脚本已验证
- 配置项已区分环境
- 错误码与前端映射已对齐

### 6.2 测试检查

- P0 缺陷清零
- P1 缺陷已评估并可接受
- 核心回归通过

### 6.3 产品与运营检查

- 文案和规则已确认
- Banner 和推荐位已配置
- 举报处理 SOP 已准备
- 灰度城市和种子用户范围已确认

## 7. 上线后观察指标

首周重点看：

- 登录成功率
- 资料完善率
- 活动报名转化率
- 到场率
- 爽约率
- 举报率
- 支付成功率

## 8. 结论

USport 的验收重点不是静态页面是否展示正确，而是“成局链路是否稳定、履约状态是否可信、治理能力是否足够”。测试必须围绕这三件事展开。
