## 2. 产品范围

### 2.1 产品目标

USport 是一个帮助用户在同城快速找到运动搭子、加入或发起运动局、完成线下成局并沉淀长期运动关系的平台。

### 2.2 当前版本范围

本说明书覆盖 `MVP + MVP 必需的运营后台能力`。

当前版本重点覆盖以下业务域：

- 账号与身份
- 运动档案
- 搭子发现与筛选
- 一对一邀约
- 公开组局 / 私密组局
- 官方活动
- 报名、候补、退出、取消
- 聊天与通知
- 到场确认与完赛
- 评价、举报、信用分
- 会员与基础支付
- 运营后台

### 2.3 当前版本不覆盖

- 教练课程平台
- 重内容社区
- 装备电商
- 场馆 SaaS
- 复杂赛事系统
- 跨城旅游型运动产品

## 3. 角色定义与权限

### 3.1 角色列表

#### R1 游客

未登录用户。

#### R2 注册用户

已登录、已完成基础注册，但未完成实名认证。

#### R3 认证用户

已完成实名认证或平台认可的可信身份认证。

#### R4 局主

可以发起运动局的用户。默认注册用户即可成为局主，但部分高风险能力需认证后开放。

#### R5 官方运营

平台内部人员，可发布官方活动、审核资料、处理举报、配置运营规则。

#### R6 客服/风控

平台内部人员，可处理投诉、封禁、信用申诉、人工审核。

### 3.2 权限总则

| 能力 | 游客 | 注册用户 | 认证用户 | 局主 | 官方运营 |
| --- | --- | --- | --- | --- | --- |
| 浏览公开首页 | 是 | 是 | 是 | 是 | 是 |
| 查看公开活动列表 | 是 | 是 | 是 | 是 | 是 |
| 发起一对一邀约 | 否 | 是 | 是 | 是 | 否 |
| 创建公开运动局 | 否 | 是 | 是 | 是 | 是 |
| 创建高风险时段/女性敏感场景活动 | 否 | 否 | 是 | 是 | 是 |
| 发起官方群活动 | 否 | 否 | 否 | 否 | 是 |
| 私聊 | 否 | 是 | 是 | 是 | 否 |

### 3.3 权限规则

1. 游客只能浏览，不可发起任何互动。
2. 注册用户可发起基础邀约与基础组局。
3. 女性友好模式、仅认证可见、夜间高风险活动等能力默认要求发起者为认证用户。
4. 官方活动只能由运营后台创建。
5. 风控能力必须保留完整操作日志。

## 5. 全局业务规则

### 5.1 城市与地理规则

1. 用户必须选择一个当前常驻城市。
2. 默认只展示当前城市内容。
3. 允许切换查看其他城市，但不可跨城报名需要线下即时履约的活动，除非发起人明确允许跨城用户。
4. 未报名通过前，活动地点仅展示模糊区域，不展示完整门牌。

### 5.2 时间规则

1. 所有活动必须有明确开始时间。
2. 开始时间必须晚于当前时间至少 30 分钟。
3. 报名截止时间不得晚于活动开始时间。
4. 活动结束后 24 小时内允许确认到场与评价，超时自动关闭评价入口。

### 5.3 信用规则

1. 新用户默认信用分为 100 分。
2. 爽约、迟到、被举报成立会扣减信用分。
3. 连续多次爽约的用户将被限制发起活动与邀约。
4. 信用分低于阈值的用户，推荐权重降低。

### 5.4 安全规则

1. 女性用户可选择仅接收认证用户邀约。
2. 夜间活动、私人场所活动、仅限异性等高风险场景需触发额外风控。
3. 聊天内容触发敏感词时，系统需自动拦截或进入人工审核队列。

### 5.5 收费规则

1. 当前版本核心匹配能力默认免费。
2. 会员主要提供高级筛选、曝光增强、数据权益。
3. 官方活动允许收取服务费。
4. 用户自发活动默认不强制平台抽佣。

## 6. 状态机定义

### 6.1 用户认证状态

- `UNVERIFIED` 未认证
- `PENDING` 认证中
- `VERIFIED` 已认证
- `REJECTED` 认证失败

流转规则：

- 用户提交认证材料后，从 `UNVERIFIED -> PENDING`
- 审核通过后，从 `PENDING -> VERIFIED`
- 审核拒绝后，从 `PENDING -> REJECTED`
- 用户补充材料后，可再次进入 `PENDING`

### 6.2 邀约状态

- `PENDING` 待处理
- `ACCEPTED` 已接受
- `REJECTED` 已拒绝
- `CANCELLED` 已取消
- `EXPIRED` 已过期
- `COMPLETED` 已完成

规则：

1. 邀约默认 24 小时有效。
2. 发起方可在被处理前取消。
3. 超过有效时间未处理自动过期。

### 6.3 活动状态

- `DRAFT` 草稿
- `PUBLISHED` 已发布
- `FULL` 已满员
- `ONGOING` 进行中
- `COMPLETED` 已完成
- `CANCELLED` 已取消
- `CLOSED` 已关闭

规则：

1. 只有已发布活动可被搜索和报名。
2. 满员后进入 `FULL`，若有人退出可回到 `PUBLISHED`。
3. 到达开始时间后自动进入 `ONGOING`。
4. 活动结束且结算完成后进入 `COMPLETED`。

### 6.4 报名状态

- `APPLIED` 已报名待确认
- `APPROVED` 报名成功
- `WAITLISTED` 候补中
- `REJECTED` 被拒绝
- `CANCELLED_BY_USER` 用户退出
- `CANCELLED_BY_HOST` 局主取消
- `CHECKED_IN` 已到场
- `NO_SHOW` 未到场
- `FINISHED` 已完成

### 6.5 举报状态

- `SUBMITTED` 已提交
- `UNDER_REVIEW` 审核中
- `RESOLVED_VALID` 举报成立
- `RESOLVED_INVALID` 举报不成立
- `CLOSED` 已关闭

## 7.1 场景 A：用户注册登录

### 场景编号

`UC-ACC-001`

### 场景目标

用户以最低门槛完成注册并进入平台。

### 触发条件

用户首次进入平台并点击登录。

### 前置条件

- 用户未登录
- 平台服务正常

### 主流程

1. 用户点击“微信登录”或“手机号登录”。
2. 系统获取用户授权。
3. 系统校验登录凭证合法性。
4. 若为首次登录，系统创建新用户。
5. 系统返回 token 与基础用户信息。
6. 用户进入新手引导流程。

### 异常流程

#### A1 授权失败

- 用户拒绝授权或授权超时
- 系统提示“登录未完成，请重试”
- 不创建用户

#### A2 凭证失效

- 系统提示登录失败
- 允许用户重新发起登录

### 业务规则

1. 若账号已存在，不重复创建新用户。
2. 同一手机号/同一微信身份只能绑定一个主账号。
3. 登录成功后必须生成可续期的会话 token。

### 输出结果

- 用户会话建立成功
- 新用户进入引导
- 老用户进入首页

### 验收标准

1. 登录成功后 1 秒内跳转首页或引导页。
2. 首次登录不会创建重复账号。
3. 登录失败时给出明确错误提示。

## 7.2 场景 B：新手引导与运动档案初始化

### 场景编号

`UC-ACC-002`

### 场景目标

让用户尽快建立可用于匹配的运动档案。

### 前置条件

- 用户已完成注册登录
- 用户未完成初始档案

### 主流程

1. 系统要求用户选择当前城市。
2. 用户选择 1-3 个主运动项目。
3. 用户为每个主运动设置技能等级。
4. 用户选择常活动区域。
5. 用户选择常活动时段。
6. 用户上传头像并填写昵称。
7. 系统保存档案并进入首页。

### 异常流程

#### B1 用户跳过

- 用户可跳过非必填项
- 但若必填项缺失，系统不得进入推荐结果页

### 业务规则

#### 必填项

- 城市
- 至少 1 个主运动
- 至少 1 个技能等级
- 常活动区域
- 昵称

#### 非必填项

- 个性签名
- 次运动项目
- 详细介绍

### 验收标准

1. 未完成必填档案时，不得发起邀约和创建活动。
2. 档案完成后，系统立即生成第一批推荐。

## 7.3 场景 C：实名认证

### 场景编号

`UC-TRUST-001`

### 场景目标

建立基础信任并解锁部分高级能力。

### 前置条件

- 用户已登录

### 主流程

1. 用户进入认证页面。
2. 系统展示认证说明、用途和隐私规则。
3. 用户提交真实姓名、证件信息或平台认可的第三方认证材料。
4. 系统进入审核状态。
5. 审核通过后，用户状态变为已认证。

### 异常流程

#### C1 审核拒绝

- 系统返回拒绝原因
- 用户可补充材料后再次提交

### 业务规则

1. 认证失败原因必须可被用户理解。
2. 认证记录必须可审计。
3. 已认证身份被怀疑异常时，平台可撤销认证状态。

### 验收标准

1. 已认证用户在资料页显示明确标识。
2. 需要认证的功能在未认证状态下显示限制原因。

## 7.4 场景 D：浏览同城搭子推荐

### 场景编号

`UC-MATCH-001`

### 场景目标

用户快速看到适合邀约的同城运动用户。

### 前置条件

- 用户已登录
- 用户完成基础档案

### 主流程

1. 用户进入“发现”页。
2. 系统按城市、运动项目、技能、距离、活跃度排序推荐用户。
3. 用户浏览推荐列表。
4. 用户可进入用户详情页查看更多信息。
5. 用户可发起邀约或收藏该用户。

### 列表展示字段

- 头像
- 昵称
- 主运动
- 技能等级
- 距离/区域
- 最近活跃时间
- 认证状态
- 信用等级

### 业务规则

1. 默认优先展示同城、同运动、技能相近用户。
2. 已被拉黑、已封禁用户不得出现在推荐中。
3. 女性用户在安全模式下，仅展示符合安全条件的对象。

### 异常流程

#### D1 无推荐结果

- 系统展示“当前筛选条件下暂无合适搭子”
- 推荐用户放宽筛选
- 推荐加入公开活动

### 验收标准

1. 推荐列表首屏加载时间不超过 2 秒。
2. 无结果时提供明确引导，不出现空白页。

## 7.6 场景 F：发起一对一邀约

### 场景编号

`UC-MATCH-003`

### 场景目标

用户向另一位用户发起一次明确的运动邀约。

### 前置条件

- 发起人已登录
- 发起人档案完整
- 对方未拉黑发起人
- 发起人未被限制互动

### 主流程

1. 用户进入目标用户资料页。
2. 点击“发起邀约”。
3. 选择运动类型、期望时间、期望区域。
4. 填写邀约说明。
5. 系统校验频率限制和安全规则。
6. 系统创建邀约记录，状态为 `PENDING`。
7. 接收方收到通知。

### 异常流程

#### F1 发送频率超限

- 系统提示“今日邀约次数已达上限”

#### F2 被对方屏蔽

- 系统提示“当前无法发起邀约”

### 业务规则

1. 同一发起人对同一接收人，在 24 小时内最多存在 1 个待处理邀约。
2. 邀约默认 24 小时过期。
3. 发起人与接收人已有黑名单关系时，不允许发起。
4. 若接收人开启“仅认证用户可邀约”，未认证发起人不得发送。

### 验收标准

1. 邀约成功后，发起页显示“待对方回应”。
2. 接收方在消息中心可见邀约卡片。

## 7.7 场景 G：处理一对一邀约

### 场景编号

`UC-MATCH-004`

### 主流程

1. 接收方收到邀约通知。
2. 接收方查看邀约详情。
3. 接收方选择接受、拒绝或忽略。
4. 系统更新邀约状态。
5. 若接受，自动建立私聊会话。

### 业务规则

1. 接受后，双方可在私聊中继续确认细节。
2. 拒绝后，不自动重发。
3. 忽略不等于拒绝，过期后系统变更为 `EXPIRED`。

### 验收标准

1. 双方都能看到邀约状态变更。
2. 接受后立即可聊天。

## 7.8 场景 H：创建公开运动局

### 场景编号

`UC-ACT-001`

### 场景目标

局主发起一场可供多人报名的公开活动。

### 前置条件

- 用户已登录
- 用户档案完整
- 用户信用分不低于活动发起阈值

### 活动创建必填字段

- 活动标题
- 运动类型
- 开始时间
- 报名截止时间
- 活动地点
- 人数上限
- 技能要求
- 是否收费
- 可见范围

### 主流程

1. 用户点击“发起活动”。
2. 填写活动信息。
3. 系统校验时间、人数、规则合法性。
4. 用户提交成功。
5. 系统创建活动并发布，状态为 `PUBLISHED`。

### 异常流程

#### H1 时间非法

- 报名截止时间晚于开始时间
- 系统阻止提交并提示修正

#### H2 发起资格不足

- 信用分过低或账号受限
- 系统拒绝发起

### 业务规则

1. 人数上限最少为 2，最多为 100，具体按运动项目配置。
2. 默认报名方式为“先到先得”。
3. 局主可选择“需审核”模式。
4. 高风险活动需触发平台审核后才可发布。

### 验收标准

1. 活动发布后可在列表中被搜索到。
2. 活动页展示核心信息完整，不得缺少时间、地点、人数、费用。

## 7.9 场景 I：创建私密运动局

### 场景编号

`UC-ACT-002`

### 场景目标

局主创建仅限受邀用户可见的私密活动。

### 业务规则

1. 私密活动不进入公开推荐流。
2. 仅被邀请者可查看详情。
3. 私密活动同样适用安全、信用、举报规则。

## 7.10 场景 J：报名公开活动

### 场景编号

`UC-ACT-003`

### 前置条件

- 用户已登录
- 活动状态为 `PUBLISHED` 或 `FULL`
- 用户未被活动发起人拉黑
- 用户未重复报名

### 主流程

1. 用户进入活动详情页。
2. 点击“报名”。
3. 系统校验是否满足报名条件。
4. 若活动为先到先得且未满员，则报名状态为 `APPROVED`。
5. 若活动已满员，则进入 `WAITLISTED`。
6. 用户收到结果通知。

### 异常流程

#### J1 技能不匹配

- 局主设置了技能门槛
- 用户不满足条件
- 系统提示无法报名

#### J2 报名截止

- 系统阻止报名

### 业务规则

1. 同一用户只能报名一次。
2. 已报名用户再次点击报名，应直接进入报名详情。
3. 若活动要求审核，状态应为 `APPLIED`，待局主处理。

### 验收标准

1. 报名后状态立即反馈。
2. 候补用户能明确看到自己处于候补中。

---

## 7.11 场景 K：局主审核报名

### 场景编号

`UC-ACT-004`

### 主流程

1. 局主进入活动管理页。
2. 查看待审核报名名单。
3. 查看报名用户档案、认证状态、信用分。
4. 选择通过或拒绝。
5. 系统更新报名状态并通知用户。

### 业务规则

1. 通过后若人数达到上限，活动状态变为 `FULL`。
2. 拒绝需填写简要原因，供用户查看。

---

## 7.12 场景 L：用户退出活动

### 场景编号

`UC-ACT-005`

### 主流程

1. 用户进入已报名活动详情。
2. 点击“退出活动”。
3. 系统根据当前时间与规则判断是否允许退出。
4. 若允许，报名状态变为 `CANCELLED_BY_USER`。
5. 若有候补用户，系统自动递补。

### 业务规则

1. 活动开始前 2 小时内退出视为临时退出。
2. 活动开始后未到场且未提前退出视为爽约。
3. 官方收费活动若涉及退款，需按退款规则执行。

---

## 7.13 场景 M：局主取消活动

### 场景编号

`UC-ACT-006`

### 主流程

1. 局主进入活动管理页。
2. 点击“取消活动”。
3. 填写取消原因。
4. 系统将活动状态更新为 `CANCELLED`。
5. 所有报名用户收到通知。

### 业务规则

1. 活动开始前可取消。
2. 若频繁取消活动，局主信用分下降。
3. 官方收费活动取消需触发退款流程。

---

## 7.14 场景 N：候补递补

### 场景编号

`UC-ACT-007`

### 主流程

1. 活动已有候补用户。
2. 正式用户退出后释放名额。
3. 系统按候补顺序递补第一位候补用户。
4. 候补用户状态变为 `APPROVED`。
5. 用户收到通知。

### 业务规则

1. 候补顺序默认按进入候补时间排序。
2. 若候补用户在递补通知后 30 分钟内未确认，平台可直接完成自动转正，不要求二次确认。

---

## 7.15 场景 O：活动群聊与私聊

### 场景编号

`UC-CHAT-001`

### 目标

让用户围绕成局进行必要沟通。

### 主流程

1. 一对一邀约接受后建立私聊。
2. 活动报名成功后自动加入活动群聊。
3. 用户可发送文本、位置、快捷确认语。

### 业务规则

1. 未报名成功用户不得进入活动群聊。
2. 活动取消后，群聊进入只读状态。
3. 敏感词消息需被拦截或打标。

### 验收标准

1. 活动群成员与报名名单保持一致。
2. 被踢出或退出活动的用户不得继续发送活动群消息。

## 7.16 场景 P：活动开始前提醒

### 场景编号

`UC-NOTI-001`

### 目标

降低忘记参加与爽约率。

### 主流程

1. 系统在活动开始前 24 小时发送首次提醒。
2. 系统在开始前 2 小时发送第二次提醒。
3. 系统在开始前 30 分钟发送最终提醒。

### 规则

1. 用户可关闭部分营销通知，但不可关闭核心履约通知。
2. 若活动时间变更，所有提醒任务需重算。

## 7.17 场景 Q：到场确认

### 场景编号

`UC-FULFILL-001`

### 主流程

1. 活动进入 `ONGOING` 状态。
2. 局主可手动确认到场名单。
3. 用户也可自助点击“我已到场”。
4. 若双方确认一致，则报名状态变为 `CHECKED_IN`。

### 业务规则

1. 到场确认时间窗为活动开始前 30 分钟至开始后 2 小时。
2. 超出时间窗未确认，系统根据规则转入待判定。
3. 若局主与用户确认冲突，进入客服复核。

## 7.18 场景 R：完赛与评价

### 场景编号

`UC-FULFILL-002`

### 主流程

1. 活动结束后，系统提示用户评价。
2. 用户可评价活动组织、准时性、沟通体验。
3. 用户可评价是否愿意再次一起运动。
4. 系统根据评价与到场情况调整信用。

### 业务规则

1. 只有报名成功且到场用户可评价。
2. 评价窗口为活动结束后 24 小时。
3. 差评不得直接公开显示对方隐私信息。

## 7.19 场景 S：爽约判定

### 场景编号

`UC-CREDIT-001`

### 判定条件

若用户满足以下条件之一，可判定为爽约：

1. 活动开始前未退出，活动期间未到场确认。
2. 局主提交未到场名单，且用户无异议或异议不成立。
3. 一对一邀约已接受，但明确约定后无故未出现。

### 系统处理

1. 报名状态变为 `NO_SHOW`。
2. 记录信用扣分。
3. 向用户发送说明与申诉入口。

### 申诉规则

1. 用户可在 24 小时内申诉。
2. 申诉需上传证据。
3. 客服审核后可撤销或维持判定。

## 7.20 场景 T：举报用户/活动/消息

### 场景编号

`UC-SAFE-001`

### 举报对象

- 用户
- 活动
- 消息
- 群聊内容

### 主流程

1. 用户点击举报入口。
2. 选择举报类型。
3. 填写举报说明并上传证据。
4. 系统生成举报工单。
5. 风控审核并给出处置结果。

### 举报类型

- 骚扰
- 虚假信息
- 违规收费
- 色情低俗
- 欺诈
- 线下安全风险
- 爽约争议

### 业务规则

1. 举报人信息默认不向被举报人暴露。
2. 举报成立后，应同步影响信用分或账号状态。
3. 严重举报可立即触发临时限制。

## 10. 风控规则细化

### 10.1 自动风控触发条件

以下情况需进入风控监测：

1. 用户短时间大量发送邀约
2. 同一设备创建多个账号
3. 消息中包含联系方式外导、色情、辱骂、诈骗词汇
4. 高频被举报
5. 高频取消活动

### 10.2 风控处置策略

- 弱提示
- 临时限制
- 进入人工审核
- 下线内容
- 封禁账号

## 11. 非功能需求

### 11.1 性能

1. 首页首屏接口 P95 响应时间小于 800ms。
2. 推荐列表接口 P95 响应时间小于 1200ms。
3. 聊天消息推送端到端延迟小于 3 秒。

### 11.2 可用性

1. 核心服务月可用性不低于 99.9%。
2. 活动、报名、消息、登录是 P0 服务。

### 11.3 安全

1. 全量接口必须鉴权。
2. 隐私数据必须加密存储或脱敏展示。
3. 后台操作必须具备审计日志。

### 11.4 合规

1. 明示用户协议与隐私政策。
2. 实名信息仅用于必要业务目的。
3. 举报与封禁需有申诉路径。
